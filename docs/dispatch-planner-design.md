# Дизайн: суточный capacity-планировщик рассылки

**Статус:** черновик v1 (2026-07-09). Согласовано: каденс — непрерывный; рендер HTML — ленивый; цель заявки — офферы (мин. 4) с режимом max_reach. Открытые вопросы — в конце.

## 1. Проблема (зачем)

Сейчас **генератор жадный**: `emails:generate-queue` сразу рендерит HTML на всех поставщиков батча в `email_queue(pending)`. **Диспетчер душит** пост-фактум (лимит ящика + пейсинг + дневной потолок получателя). Разница копится в `pending` → **пробка**. Волны v2, held-пул и 4 реактивных деферрала (`deferred_batches`: sender_capacity / recipient_load / quality-gate, + followup) — костыльное приближение планирования; система борется сама с собой.

**Первопричина ожога доменов** (см. `memory/iqot-deliverability-regression.md`): на хендовере n8n→Laravel сняты неявные рейт-лимиты → бёрст скорости домена (1680/час) + вал на получателя (117/день с 82 ящиков → FBL-жалобы). Ёмкость должна быть **входом планирования**, а не тормозом.

## 2. Ключевая идея — разделить BACKLOG и OUTBOX

| Уровень | Что | Стоимость | Живёт |
|---|---|---|---|
| **Backlog спроса** | «интент»: кому хотим написать по какой заявке (request×supplier), с рангом/тиром | дёшево (строка) | копится сколько угодно, не протухает |
| **Outbox** | отрендеренное письмо, готовое уйти (`email_queue`) | дорого (AI-тело + HTML) | рендерим **лениво**, только под то, что влезает в ёмкость |

Из всего редизайна это — главное. Пробка исчезает by design: рендерим ровно то, что уйдёт. Волны/приоритет/обратная связь становятся *следствием*, а не отдельными механизмами.

## 3. Сущности и ёмкости

**Единица работы — ПОЗИЦИЯ (`request_item`), НЕ батч.** Мы не ограничены батчами: ведём список активных позиций. У каждой — своя цель покрытия и своё состояние. Позиция, закрытая «с лихвой», **выпадает из работы глобально** → освобождает ёмкость получателей/сендеров под другие позиции (нет смысла её дальше слать).

**Спрос:** `send_intents` — гранулярность **(request × supplier)**, но payload = АКТИВНЫЕ позиции этой заявки, матчащие поставщика (на момент рендера). Поля: request_id, supplier_id, tier (hot/warm/cold), score, status (backlog|rendered|sent|dropped). Источник: подбор пула (`CampaignSupplierSelector`) + discovery. Тир = качество матча (Яндекс-hot → warm-добор → cold-пул).

**Батчей упреждающих нет.** «Батч ≤5» — только ограничение РАЗМЕРА письма: на рендере берём активные позиции поставщика и, если их >5, режем на несколько писем. Список позиций динамический (только ещё-не-закрытые) → закрытая позиция автоматически перестаёт попадать в новые письма.

**Ёмкости (на день, МСК):**
- **Сендер:** `SenderDailyCapacity` — `warmup daily_limit − израсходовано` (переиспользуем как есть).
- **Получатель:** адаптивный дневной cap (см. §5). Anchor — `recipient_mailboxes` (уже есть `daily_sent_count/date`).
- (опц.) **Глобальный потолок** платформы/сутки.

## 4. Формула приоритета (score интента)

```
score(intent) =
    w_deficit  * deficit_factor(request)      // главный драйвер
  + w_customer * is_customer_request
  + w_age      * age_factor(request)          // старые не должны голодать
  + w_tier     * tier_weight(hot|warm|cold)   // hot вперёд
  + w_rating   * supplier_rating_norm
```

**Покрытие считается ПО ПОЗИЦИИ** (не по заявке в среднем). Позиция **satisfied**, когда набрала ≥ `offer_target` (мин. 4) РАЗНЫХ ценовых ответов (переиспользуем логику `coverageForBatch` → на уровень позиции). Satisfied-позиция выпадает из активного набора → её больше не включаем в письма.

**deficit_factor(request)** = агрегат по ещё-активным позициям заявки:
- **target** (дефолт, `offer_target≥4`): по каждой активной позиции `deficit_i = max(0, offer_target − offers_i)`; фактор заявки `= (Σ deficit_i / Σ offer_target)` по активным позициям. Заявка полностью **satisfied**, когда активных позиций не осталось (все закрыты) — интенты выбывают.
- **max_reach** (флаг заявки): фактор держится высоким (константа), пока **пул кандидатов не исчерпан** — шлём всем, не останавливаясь на 4; позиции не «закрываются» досрочно.

Интент (request×supplier) «живой», пока у поставщика есть ≥1 активная матчащая позиция, ещё не отправленная ему.

«Волны» = следствие: сначала в план проходят hot-интенты (высший tier_weight); заявка недобрала → на следующих тиках проходят warm/cold того же реквеста. Никакого held-пула и таймерных волн.

## 5. Адаптивный дневной cap получателя

- База **10**/день (в т.ч. для нового адреса без истории). Позитивный сигнал (ответил / прислал оффер, нет баунсов) → рост **до 15** постепенно. Отсутствие реакции / рост баунсов → спад **до 5**. Жёсткий баунс/жалоба → блок (`is_blocked`, уже есть).
- Хранение: расширяем `recipient_mailboxes` (текущий `daily_sent_count/date`) полями `cap_current`, сигналы вовлечённости. Пересчёт — в обратной связи (§7).

## 6. Планировщик (непрерывный top-up)

Каждый тик (диспетчер и так поминутно):
1. Посчитать остаток ёмкости: по каждому сендеру и получателю на сегодня (+ глоб. потолок).
2. Взять из backlog топ-N интентов по `score`, у которых **satisfied=false**.
3. Жадно назначить: для интента — если у его получателя есть остаток дневного cap И есть подходящий сендер (правило org/анти-фингерпринт) с остатком → **отрендерить письмо (лениво)** и положить в `email_queue(pending, scheduled=now)`. Иначе — пропустить, интент остаётся в backlog.
4. Дальше — существующий диспетчер (пейсинг/слоты/отправка) без изменений.

«Суточный план» = *приоритет + бюджет ёмкости*; **исполнение непрерывное**. Экспресс-полоса: клиентские/срочные заявки — повышенный `w_customer` (проходят вперёд), опц. в обход накопления.

Ленивый рендер: тело батча (greeting/intro/ai_request, `CampaignBodyGenerator`) генерим 1 раз на батч и кешируем; per-supplier `CampaignEmailBuilder` — дёшево, на момент выбора интента.

## 7. Обратная связь (утро/непрерывно)

- **Доставка:** баунсы/спам-реджект по получателю → снижаем его cap / блок; по домену-отправителю → сигнал притормозить/отлежаться.
- **Вовлечённость:** ответ/оффер → поднимаем cap получателя (до 15).
- **Цель заявки:** пересчёт `offers_received` + позиционного покрытия → satisfied → интенты выбывают (target); max_reach — до исчерпания пула.
- **Приоритеты:** недобравшие заявки всплывают (deficit растёт с возрастом).

## 8. Что переиспользуем / ретайрим

**Переиспользуем:** `SenderDailyCapacity`, `CampaignSupplierSelector`/ранжирование, discovery, `CampaignBodyGenerator`+`CampaignEmailBuilder`, `recipient_mailboxes` + дневной cap, весь низ диспетчера (пейсинг/слоты/анти-бан).

**Ретайрим (сливается в планировщик):** waves_v2 + held-пул, `deferred_batches` (load/capacity/quality), `ProcessLoadDeferred`/`ProcessCapacityDeferred`/`RetryDeferred`, `DispatchFollowupEmails`, `WarmupBatchSplitter`.

**Предусловие:** сначала стабилизировать репутацию (DKIM/DMARC + отлёжка сгоревших) — иначе эффект планировщика не измерить (смазано горелыми доменами).

## 10. Решения и открытые вопросы

**Решено (2026-07-09):**
- Каденс — **непрерывный** top-up к дневному бюджету ёмкости.
- Рендер HTML — **ленивый**, под запланированное.
- Гранулярность — **(request × supplier)**; покрытие и выпадение — **ПО ПОЗИЦИИ**; батчи ≤5 только как размер письма на рендере (не упреждающие).
- Стартовый cap нового адреса — **10**.
- Экспресс клиентских — **весом** (`w_customer`), без отдельного тракта.

**Ещё открыто:**
1. **Глобальный суточный потолок** платформы — нужен ли поверх пер-сендер/пер-получатель (страховка)?
2. **Веса** `w_*` в score — стартовые значения и где хранить (config vs таблица настроек).
3. **`offer_target` по умолчанию** — 4 глобально, или настраиваемо на заявку/категорию (плюс флаг `max_reach`).

# МОДЕЛЬ v2 — ДНЕВНОЙ ПЛАНИРОВЩИК + TOP-UP (согласовано 2026-07-10, ЦЕЛЕВАЯ)

v1 (непрерывный top-up) близорук: рендерит «здесь и сейчас» малыми данными → 93% батчей = 1 позиция, поставщик с несколькими позициями получает N одиночных писем. v2 планирует **на дневное окно** сразу по всей потребности и всем ресурсам.

## v2.1 Принципы
- **Раз в день** (утро, до окна) — тяжёлый прогон строит **план дня**. В течение дня план не пересобирается.
- **Единица письма = поставщик + ВСЕ его совместимые активные позиции** (не 1 позиция). Одно письмо на поставщика в день.
- **top-up в течение дня** — только добор появившихся из discovery НОВЫХ поставщиков по ещё-открытым позициям. Новые ЗАЯВКИ, пришедшие днём, ждут утра (лимиты дня расписаны + ждём заявки на совмещение позиций).

## v2.2 Утренний прогон (по шагам)
1. **Активные позиции** = все `request_item` с `офферов < offer_target` (новые + «на добор» — одно множество). Заявки completed/cancelled исключены.
2. **Пул на позицию (редуцированный):** профильные поставщики `− уже писавшие ПО ЭТОЙ ПОЗИЦИИ` (по `request_item_responses`: строка = поставщику писали). Позиция **выпадает**, если набрала target ЛИБО пул исчерпан (всем писали).
3. **Яндекс-таргетинг:** на позицию — **приоритетные** (у кого нашлась страница = релевантные) из остатка пула + новые домены → discovery.
4. **AI-аффинность (1 прогон на всю очередь):** для каждой позиции — с какими другими она **совместима** в одном письме / какие **не мешать**. Мягкая матрица, НЕ жёсткие группы.
5. **Назначатель (жадный, ядро):**
   - **Фаза приоритета:** идём по парам «позиция → её релевантный поставщик» по срочности (дефицит+клиент+возраст). На поставщика собираем ВСЕ его релевантные активные позиции, бандлим по аффинности (совместимые вместе, ≤N в письме; несовместимые → отдельное письмо) → письмо **в начало дня**.
   - **Фаза добора:** позициям, не набравшим target релевантными, добираем из общего пула тем же бандлингом — заполняя остаток дневной ёмкости.
   - **Ограничители:** поставщик ≤ его дневного cap (в идеале 1 письмо); позиция — стоп при target (кроме max_reach); ящики round-robin под warmup-ёмкость.
   - Результат = **упорядоченный план дня** (кому / какие позиции / с какого ящика / очерёдность).
6. Рендер плана лениво под ёмкость (переиспуя render-путь) → `email_queue`; дедуп `request_item_responses` на позицию×поставщика.

## v2.3 Дневной цикл (замкнутый)
Утро (план) → день (рассылка по плану + discovery + top-up новых поставщиков) → ночь (пришли КП: позиции добрали/закрылись; накопились новые заявки) → следующее утро (п.1 с обновлённым состоянием, редуцированные пулы).

## v2.4 Что меняем vs v1 (уже построенного)
- **Дедуп на уровень ПОЗИЦИИ** (сейчас на уровень заявки в `build-intents`): поставщик, писавший про позицию A, может получить письмо про новую активную позицию B, которую матчит. `PositionCoverage` расширить: emailed-suppliers per position, remaining-pool, isExhausted.
- **AI-аффинность** — новый сервис (1 вызов на очередь).
- **Назначатель** — новый сервис (жадный, ядро). Заменяет наивный per-product-type грайндер в `plan-render`.
- **Утренний прогон** — новая команда `emails:plan-day` (раз в сутки), делает шаги 1-6.
- **top-up** — `plan-render` переводим в режим «только добор новых поставщиков по открытым позициям».

## v2.5 Порядок реализации (за флагом, top-up v1 в бою пока)
1. `PositionCoverage` → позиционный дедуп/остаток пула/исчерпание.
2. `PositionAffinity` (AI, 1 прогон).
3. `DayPlanAssigner` (назначатель, ядро) + `emails:plan-day`.
4. Утренний прогон с Яндекс/discovery.
5. Переключить `plan-render` в режим top-up-новых.
